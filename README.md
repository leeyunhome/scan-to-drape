# Scan-to-Drape — a native 3D Gaussian Splatting viewer, built toward draping cloth on a real scan

A C++/OpenGL viewer that loads a 3D Gaussian Splatting reconstruction straight from the trainer's
PLY output and rasterizes it with the same EWA-splat projection math as the reference renderer —
no Qt/OpenGL/Vulkan library did this for me; this is the from-scratch native counterpart to the
browser-based [splatting-viewer](../splatting-viewer) project, aimed at a further goal: draping a
simulated cloth mesh over the scan itself.

## What's here (Phases 1, 1.6, 2, 3 of the roadmap)

- `include/gsplat/ply_loader.hpp` — parses the standard INRIA 3DGS binary PLY export (position,
  log-scale, quaternion rotation, opacity logit, DC-term spherical-harmonic color) by property
  name, so it isn't locked to one fixed SH degree.
- `shaders/splat.vert` / `splat.frag` — projects each splat's 3D covariance to a 2D screen-space
  covariance via the Jacobian of the perspective projection, then rasterizes an oriented quad
  sized from the eigen-decomposition of that covariance, with a true anisotropic Gaussian falloff
  evaluated in the fragment shader (not a fixed-radius circle).
- `include/gsplat/mesh.hpp`, `shaders/mesh.vert` / `mesh.frag` — a small procedural-sphere generator
  and Phong shader, used for the Phase 2 depth-compositing test mesh and to render the Phase 3 cloth.
- `include/cloth/cloth.hpp` — vendored from [cloth-simulation](../cloth-simulation) (Phase 3).
- `tools/clean_scan.py` — offline outlier cleanup for a raw 3DGS PLY export (Phase 1.6).
- `src/main.cpp` — GLFW + glad window/context, an orbit camera, a per-frame CPU depth sort so
  alpha-blended splats composite back-to-front, the two-pass mesh/splat depth compositing, and the
  Phase 3 collision-proxy extraction + cloth simulation loop.

## Verified

Ran against the real `mechander.ply` scan (386k splats after filtering, from
[splatting-viewer/models](../splatting-viewer/models)) and confirmed by screenshot
(`notes/mechander_v0.png`) that the pipeline — load → project → sort → blend — produces a
recognizable, correctly-shaded reconstruction. Interactive orbit/zoom via left-drag and scroll.

**Known limitation, found and diagnosed, not hidden:** the raw training checkpoint (a 26-photo,
sparse-view capture) has a long tail of oversized, often flat or needle-shaped "floater" splats —
a well-known failure mode of few-view 3DGS. The v0 loader's `--keep-percentile` (a single global
scale cutoff) is a stand-in for real cleanup, not the fix — see Phase 1.6 below for what actually
helped.

**Phase 1.6 — offline cleanup, `tools/clean_scan.py`.** Compared four outlier-detection strategies
on the gaussian centers: two from Open3D (statistical and radius outlier removal, both position-
based) and a from-scratch K-nearest-neighbor local-scale-ratio detector, plus a combination. Actually
rendered and screenshotted each, not just measured in the abstract:
- Open3D's position-based removal (statistical or radius) targets *spatially sparse* points. On
  this capture that signal doesn't line up with "floater splat" at all — large-scale floaters sit
  right next to normal-density geometry, and pulling `remove_statistical_outlier` in *removed*
  legitimate, just-sparsely-sampled surface coverage (large flat regions naturally get fewer, bigger
  splats) and tore visible holes in the model when rendered. Dropped entirely, not tuned down — the
  mismatch is about what signal it measures, not the threshold.
- The K=20-nearest-neighbor **local** scale-ratio check (flag a splat if its scale exceeds 4x its own
  neighborhood's median scale, rather than one global percentile) is the one that held up under
  rendering: removes 2.3% of splats (9,071 of 388,261) and preserves body coverage as well as the v0
  global cutoff did, while targeting the actual failure mode instead of a proxy for it.
- A background note on process, not just results: an earlier automated 4-way comparison sweep
  reported very different numbers (all four configs) than a from-scratch, self-verified rerun of the
  same math produced — tracing it down, the sweep's own scale parsing was off (its reported median/
  cutoff values didn't match this same PLY's known-good values from the C++ loader's own console
  output). Re-verified everything by hand before trusting any number that fed a decision.

`models/mechander_clean.ply` (379,190 splats, generated by `tools/clean_scan.py`) is the current
scan used for rendering; `--keep-percentile` in the viewer still exists as a coarser, in-app fallback
knob but defaults to a no-op now that cleanup happens upstream.

**Performance, measured, not assumed:** ~22 fps at 1600×1000 on an RTX 3080, for 379k splats. The
bottleneck is the naive full `std::sort` + full instance-buffer re-upload every single frame — a
known, deliberate v0 simplification. Next: throttle the resort to every few frames, or move it to
a background thread.

## Phase 2 — hybrid mesh + splat rendering

`--test-mesh` adds an opaque test sphere (`include/gsplat/mesh.hpp`, `shaders/mesh.*`) positioned
through the middle of the scan, rendered with standard Phong shading, to exercise depth compositing
in both directions: the sphere should occlude splats behind it *and* be occluded by splats in front
of it. Two-pass approach: the mesh writes both color and depth normally; the splat pass keeps depth
*testing* on but turns depth *writing* off (`glDepthMask(GL_FALSE)`), so splats behind the mesh are
correctly rejected while splat-vs-splat ordering still comes from the CPU sort, not the depth buffer.

**Bug found and fixed by direct pixel verification, not by inspection:** the first version rendered
nothing for the mesh at all — no color, no visible occlusion — despite zero GL errors and correct
uniform/attribute setup. `glReadPixels`'d the exact center pixel right after the mesh draw call
(before the splat pass, before the buffer swap) and got the plain clear color back: the mesh geometry
was never rasterizing. Root cause: `glClear(GL_DEPTH_BUFFER_BIT)` is a no-op wherever `glDepthMask`
is currently `false` — and the *previous* frame's splat pass leaves it `false` on the way out. So
from frame 2 onward, the depth buffer was never actually cleared; the mesh kept re-submitting the
same depth values against its own stale values from the prior frame, and `GL_LESS` rejected every
fragment as not-strictly-less-than itself. Fix: force `glDepthMask(GL_TRUE)` immediately before the
per-frame clear. Verified with `notes/phase2_with_mesh.png` and `notes/phase2_small_sphere.png` —
the sphere now shades correctly and both occlusion directions are visible in the same frame.

## Phase 3 — cloth drape over a proxy extracted from the scan

`--drape` vendors [cloth-simulation](../cloth-simulation)'s `cloth.hpp` (unmodified except for one
added `teleport()` — see the file), extracts a collision proxy directly from the splat cloud (the
centroid of every gaussian center, and a radius at a percentile of distance from it — see
`computeCollisionProxy` — so a single outlying limb doesn't blow up a bounding-box-based radius),
and drops an unpinned cloth grid onto that proxy under gravity, colliding via `Cloth::collideSphere`
plus a floor clamp.

**Verified it's actually simulating, not just rendering a static mesh:** logged the cloth's Y-range
every 30 frames (`notes/phase3_topdown.png` and `phase3_midfall.png` are from this same run) — it
falls smoothly from `startY≈7.56` down through the proxy sphere and settles at `floorY≈-5.74` by
around frame 300, with the min/max spread widening exactly when it's in contact with the sphere
(frames ~150–270), confirming the collision response is doing something, not passing through.

**Real limitation found, not glossed over:** `collideSphere` only pushed penetrating particles back
to the surface *radially* — it didn't touch their Verlet-implicit tangential velocity, i.e. there was
no friction. An unpinned sheet over a smooth sphere with zero friction does exactly what it should
physically do: it slides off and pools on the floor next to the proxy rather than clinging to it.
`phase3_midfall.png` (frame 175, still falling toward the object) shows the intended silhouette best;
by `phase3_topdown.png` (frame 400, pre-friction) it's already mostly slid down.

**Fixed**: `collideSphere` now takes a `friction` parameter (default `0.6` in the viewer). At contact,
it decomposes the Verlet-implicit velocity into normal/tangential components, zeroes the normal part
(inelastic — no bounce) and damps the tangential part by `friction`, instead of leaving both
untouched. `friction = 0` reproduces the exact original behavior for comparison. Verified by the same
bbox logging: without friction the cloth's max-Y collapses to floor level by frame ~390
(`y=[-5.74,-4.37]`); with `friction=0.6` it holds near the sphere's top the whole time
(`y=[-5.74,2.85]`, sphere top ≈ 3.2) — `notes/phase3_friction_v1.png` shows it actually staying up
around the object instead of fully pooling.

**Separate, unrelated issue found while hunting for a good camera angle to show this off:** at camera
pitches much above the ~15° used everywhere else in this README (tried 25°/35°/55°, at multiple
distances), the frame gets dominated by a small number of oversized splats up close — a leftover
residual of the still-imperfect Phase 1.6 cleanup (recall it removes *local* scale outliers; a few
large-but-locally-unremarkable splats survive and apparently face the camera edge-on at low pitch,
hiding their true screen-space size, but face-on at steeper pitch). Confirmed this is a splat/camera
interaction and not a new cloth bug: the cloth's own bounding box stays perfectly sane
(`x≈[-4.9,6.5]`, consistent across every pitch tested) while only the *visual* blows up. Parked as a
follow-up for Phase 1.6's cleanup rather than fixed here, since it's orthogonal to the drape work.

## Phase 1.5 — WebAssembly build (same C++ core, in the browser)

The exact same `include/gsplat/*` + `include/cloth/*` core, compiled a second way via Emscripten
(`em++`, `-s USE_GLFW=3`, WebGL2/GLES3) instead of native GLFW+glad — not a rewrite, a second build
target. `CMakeLists.txt` branches on `if (EMSCRIPTEN)`; `src/main.cpp` branches on `#ifdef
__EMSCRIPTEN__` in a handful of places (window/GL init, and swapping the native `while` loop for
`emscripten_set_main_loop_arg`, since a browser can't block a thread waiting for frames — the whole
per-frame body got pulled out of `main()` into a free `frameStep(App&)` function for exactly this,
taking an explicit `App` struct instead of closing over locals). Shader source files carry no
`#version` line anymore; `gsplat::shader.hpp` prepends `#version 330 core` or `#version 300 es` +
`precision highp float;` depending on target, so one `.vert`/`.frag` file serves both builds.

**The payload problem, actually solved, not just planned:** the raw scan is 90+ MB — too big to ship
to a browser. `tools/export_web.py` trims every splat down to only the 14 floats this renderer
reads (position, DC-term color, opacity, scale, rotation — dropping normals and the 45 unused
higher-order SH coefficients), losslessly for this specific renderer: 379,190 splats,
**94.0 MB → 21.2 MB (22.6%)**. Bundled into the WASM build via `--preload-file`.

**Built and served locally.** `cmake --build build-web` produces `viewer.html/.js/.wasm/.data`
(21.2 MB total) with zero build errors; `python -m http.server` + `curl -I` confirms all four files
respond `200` with the right content-types (`viewer.wasm` as `application/wasm`, the type browsers
need to stream-compile it). This agent has no headless-browser tool, so the actual WebGL2 render
was confirmed by a human opening the page — which immediately surfaced a real bug automation alone
wouldn't have caught (next section).

**Bug found only by opening it in an actual browser: blank canvas, no console output initially
visible.** The DevTools console (once opened) showed the real error:
`TypeError: Failed to execute 'decode' on 'TextDecoder': The provided ArrayBuffer value must not be
resizable`, thrown from inside a syscall reading a filename. Root cause: `-s ALLOW_MEMORY_GROWTH=1`
backs the WASM heap with a *resizable* `ArrayBuffer`, and current Chrome's `TextDecoder.decode()`
rejects resizable buffers outright — a real Emscripten/Chrome compatibility gap, not a bug in this
code. Fix: drop growth, use a fixed `-s INITIAL_MEMORY=268435456` (256 MB, comfortably over what
379k splats + the preloaded scan need) instead. Exactly the kind of failure mode `--screenshot`
verification can't catch — this agent's own tooling has no way to load a page and read its JS
console, so this one needed a human in the loop.

**Camera presets added, then found to be wrong, then fixed with a measurement, not a guess.**
Added front/left/right/top/bottom preset buttons plus an orientation gizmo — but "front" was
defined as world yaw = 0, and clicking it showed the same unrecognizable angle as before. The
reconstruction's world orientation is arbitrary (COLMAP has no notion of which way an object
"faces"), so there was never a reason yaw = 0 would line up with the figure's front. Fixed by
measurement instead of trial and error: computed the centroid of the figure's distinctly-colored
(red/yellow) splats — its one clear landmark — versus the whole cloud's centroid, and solved for
the yaw that puts the camera on that same side: **179.2°**. Cross-checked against a 12-angle render
sweep at 30° steps (`notes/yaw_sweep_contact_sheet.png`) — the landmark sits visibly dead-center in
frame only right around yaw = 180, and off-center at every other angle, confirming the computed
value independently. The original default view (yaw = 30) turns out to have been ~150° off-front
— effectively a back-left angle — which is exactly why nothing in this whole project's earlier
screenshots ever looked like a clearly-facing figure. Fixed both the presets (`kFrontYawDeg`) and
the app's own default starting view (a 25°-off 3/4 angle near the front, not an arbitrary one),
verified with `notes/preset_front_left_right.png`.

**Second orientation bug, same root cause: "up" was wrong too, not just "front."** Fixing yaw made
the front view legible, but the figure still rendered tilted onto its side — a real user caught
this from the live page, not a screenshot. World "up" in a COLMAP/3DGS reconstruction is exactly
as arbitrary as "front"; nothing says (0, 1, 0) has to be vertical. Measured the true up axis the
same way as before — from the scan's own geometry, not a guess: took the opacity-weighted
covariance of every splat position and eigendecomposed it. The figure stands on a flat mini-PC
case, and a thin flat surface contributes far less variance through its own thickness than in its
plane — so the *smallest*-variance eigenvector is that platform's surface normal, i.e. true up. It
came out **~68° away from world +Y**: the reconstruction isn't just yawed, it's tilted onto its
side outright. `kWorldAlign` (a fixed rotation matrix, derived once from this scan and folded into
the view matrix at render time — `view * kWorldAlign`, applied nowhere else, so cloth physics keeps
simulating in the original space) rotates that measured axis onto +Y; `kFrontYawDeg` was then
re-measured inside the now-corrected upright frame (170°, not 180° — the earlier number was only
approximately right, measured before the tilt was known about). Verified with
`notes/upright_front.png`: the figure now stands upright on its platform from every preset angle,
including top and bottom. Also tightened the default zoom per the same feedback
(`camDistFactor` 2.5 → 1.1).

## Building

Requires MSVC (Visual Studio, Desktop C++ workload) and [vcpkg](https://github.com/microsoft/vcpkg).

```bash
vcpkg install glfw3 glad glm stb --triplet x64-windows

# from a "x64 Native Tools" developer prompt:
cmake -S . -B build -G Ninja -DCMAKE_TOOLCHAIN_FILE=<vcpkg-root>/scripts/buildsystems/vcpkg.cmake
cmake --build build
```

## Running

```bash
build/viewer.exe path/to/scene.ply
build/viewer.exe path/to/scene.ply --screenshot out.png --frames 30   # headless-ish verification
build/viewer.exe path/to/scene.ply --keep-percentile 0.99             # in-app fallback outlier cut
build/viewer.exe path/to/scene.ply --test-mesh --sphere-radius-factor 0.4  # Phase 2 depth-compositing demo
build/viewer.exe path/to/scene.ply --drape --friction 0.6  # Phase 3 cloth drape (keep pitch near the default ~15°)

# clean a raw 3DGS export before viewing it (Phase 1.6):
python tools/clean_scan.py path/to/raw.ply path/to/raw_clean.ply
```

Rendering* (SIGGRAPH 2023) — the covariance-projection and EWA-splat formulas this renderer
